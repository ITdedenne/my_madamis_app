📖 My Madamis App (マダミス管理アプリ)

「マーダーミステリー」の体験（通過済）、資産（所持）、関心（購入検討・PL希望）をデジタルで一元管理し、他者と安全に共有することで、卓の成立（立卓率）を最大化するアプリケーションです。

（※ここにアプリのホーム画面や一覧画面のスクリーンショットを1〜2枚配置すると視覚的なアピールが向上します）

📱 概要 (Overview)

本アプリケーションは、マーダーミステリー愛好家が抱える「誰がどのシナリオを通過済みか分からない」「遊びたいシナリオがあるがGM（ゲームマスター）やPL（プレイヤー）が揃わない」という課題を解決するためのプラットフォームです。
ユーザー同士が繋がり（フレンズ機能）、グループでのプレイ可能シナリオの検索や、特定のシナリオに対する未通過者の絞り込みを行うことで、スムーズなゲーム開催を支援します。

🛠 使用技術 (Tech Stack)

フロントエンド: Flutter (Riverpod, MVVMアーキテクチャ)

バックエンド・インフラ: AWS Amplify (Cognito, S3, DynamoDB, Lambda)

IaC (Infrastructure as Code): Amplify CLI (overrides.ts によるCORS設定等の管理)

💡 プロジェクトのハイライト / 設計思想 (Design Philosophy)

本プロジェクトは、単なる技術の学習や機能の実装にとどまらず、「ユーザーのビジネス課題をいかに解決するか」「限られたリソース（コスト・制約）の中でいかに最適なアーキテクチャを構築するか」という、ITコンサルタント・上流設計の視点に重きを置いて開発しました。

実装作業においては生成AIを積極的に活用して徹底的に効率化を図り、私のリソースの大部分を「要件定義」「クラウドアーキテクチャ設計」「UXとコストのトレードオフ調整」といった上流工程と論理設計に投資しています。

以下に、本プロジェクトにおける具体的な課題解決のアプローチと設計の強みを記載します。

1. ビジネス課題の解決とKPI（立卓率）向上を主眼に置いた要件定義

言われた機能をそのまま作るのではなく、ユーザーの実際の利用シーンを分析し、システム側から「目的達成（ゲームの開催）」をアシストする機能を提案・実装しました。

機会損失を防ぐ「惜しい！(Near Miss)」機能: グループ検索において、単に「全員が未通過で遊べるシナリオ」だけを表示するのではなく、「あと1人メンバーを入れ替えれば遊べるシナリオ」も明確に区別して表示する仕様を定義しました。これにより、ユーザーの「あと一歩で遊べたのに」という機会損失を防ぎます。

「外部GM候補」の自動提案: 選択したメンバー内にゲームマスター（GM）可能なユーザーがいない場合でも、検索条件外のフレンズからGMを依頼できる可能性（所持・購入検討者）を自動で探索し、提案するロジックを実装しました。

【設計の意図】: これらは単なるデータベース検索ではなく、「いかに立卓率（ゲームが成立する確率）を向上させ、アプリのコアバリューを高めるか」というビジネス・KPI視点に基づく要件定義です。

2. クラウドインフラの「コスト」と「パフォーマンス」の最適化

AWSサーバーレスアーキテクチャ（Amplify）を採用するにあたり、ランニングコストを最小化しつつ、ユーザー体験を損なわないデータ設計を行いました。

静的データと動的データの分離: 高価なDynamoDBの読み取りコストを削減するため、変更頻度の低いマスターデータ（シナリオ情報等）はS3に配置し、Webブラウザからの直接アクセスを許容するCORS設定をIaCで管理しています。クライアント起動時にメモリに一括ダウンロードして処理するアーキテクチャを採用し、低コストと高レスポンスを両立させました。

クライアントサイド処理へのオフロード（計算コストの削減）: 「フレンズ内検索（名前フィルタ）」や「シナリオの人数整合性チェック」において、都度サーバー（Lambda/DynamoDB）へクエリを投げる実装を避けました。アプリの「フレンズ上限100人」というドメイン制約を逆手に取り、クライアント端末のメモリ上でリストフィルタリングを実行することで、追加のReadコストを完全にゼロに抑えています。

3. トレードオフを制する高度なUX設計と例外ハンドリング

システムの技術的制約や不整合がユーザー体験（UX）を損なわないよう、先回りしたフロントエンドの防衛的実装を行っています。

エラーの許容とUXの保護: 開発環境特有のデータ不整合等により、DBへの書き込みは成功しているもののGraphQLの戻り値でエラー（Null制約違反など）が発生するケースにおいて、システムダウンを防ぐ例外ハンドリングを定義しました。特定の許容リストに含まれるエラーはフロントエンド側で「成功」として扱い、ユーザーには正常な完了UIを提示するようコントロールしています。

APIコストを維持した画面統合とUX改善: グループ検索において、「条件設定画面」と「結果画面」を統合しました。これによりメンバー変更時の無駄な画面遷移をなくしUXを向上させつつ、「APIのコール回数は変わらないためインフラコスト増にはならない」という論理的な裏付けのもと設計変更の意思決定を行っています。

4. すべてのUI/UXに「論理的な根拠（Rationale）」を持たせる

「なんとなく見栄えが良いから」ではなく、マテリアルデザインの原則や数値的根拠に基づいたUI設計を行っています。

無限スクロールの「48件」というマジックナンバー: PCやタブレットでのグリッド表示時の初期描画制限を「48件」に設定しました。これは、画面幅の変動によって列数が「2, 3, 4, 6列」と変化しても、48がそれらの公倍数であるため、「最後の行が半端に余る」というレイアウト崩れが極めて起きにくいという数学的アプローチに基づいています。

アスペクト比「2.0」の採用: リストアイテム（カード）の比率を幅:高 = 2:1 に設定しました。画像がないテキスト主体の情報カードにおいて「上下の間延び」を防ぎつつ、長いタイトルが入った際の「BOTTOM OVERFLOWED（描画領域溢れ）」エラーを確実に回避するための、最適解としての数値を導き出した結果です。

✨ 主な機能 (Features)

認証・アカウント管理: AWS Cognitoを利用したセキュアなサインアップ/サインイン。

シナリオ手帳機能: S3から一括取得したシナリオデータに対し、「通過済」「所持」「購入検討」「PL希望」のステータスを即座に登録・管理。

フレンズ機能: 他ユーザーをフォローし、相互にマイリストを閲覧可能。

シナリオグループ検索: 選択したメンバー（自分＋フレンズ）のステータスと人数条件を掛け合わせ、「全員が遊べるシナリオ」や「惜しいシナリオ」を抽出。

プレイヤーファインダー: 特定のシナリオを指定し、それを「まだ遊んでいない（PL希望）」友人や、「GMができる（所持している）」友人を探し出すマッチング機能。

🚀 今後の展望 (Future Plans)

トランザクション管理の強化: ユーザープロフィール更新（CognitoとDynamoDBの同期）における、部分的な障害時のロールバック処理の実装。

エンゲージメント施策の拡充: 定性的な思い出を残せる「感情タグ」や「推しシナリオのピン留め」機能の追加。

バイラル（拡散）施策: アプリ内からのSNSシェア導線や、オフライン対面時を想定したQRコードによるフレンド追加機能の実装。
